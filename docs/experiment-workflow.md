# 实验版本迭代开发规范

本文档定义了实验版本迭代开发的完整流程，确保主版本稳定性的同时支持快速迭代验证。

---

## 一、目录结构

实验版本采用 Next.js App Router 路由化方案，文件组织如下：

```
pactzo/app/experiments/
├── layout.tsx              # 实验版本共用布局（顶部显示版本标识）
├── v1/
│   └── page.tsx            # 基线版本（当前 page.tsx 的精确拷贝）
├── v2/
│   └── page.tsx            # 第一次迭代
│   └── components/         # (按需) 本版本修改的子组件
└── ...

pactzo/docs/
├── modification-log.md     # 版本修改日志
└── experiment-workflow.md  # 本文档
```

**浏览器访问路径**：
- 基线版本：`/experiments/v1`
- 迭代版本：`/experiments/v2`、`/experiments/v3`、...

---

## 二、不可触碰的文件

以下文件为主版本核心文件，**禁止在迭代过程中直接修改**：

| 文件路径 | 说明 |
|---------|------|
| `app/page.tsx` | 主版本首页 |
| `app/layout.tsx` | 主版本根布局 |
| `app/globals.css` | 全局样式 |
| `components/` 下所有文件 | 共用组件目录 |

> ⚠️ 如需修改共用组件，必须走「子组件隔离流程」（见第四节）

---

## 三、迭代闭环流程

### 1. 开始前：版本状态检查

每次收到新迭代指令时，**先自动检查当前实验版本状态**：
- 最新版本号是多少
- 已有哪些改动
- 文件完整性是否正常

确认无误后再开始执行迭代任务。

### 2. 开发中：创建新版本

- **命名规则**：`experiments/vN/page.tsx`（v1, v2, v3... 依次递增）
- **起点**：复制上一版本作为基础
- **范围**：所有改动仅在新版本目录中进行
- **历史保留**：历史版本全部保留，便于回溯对比

### 3. 开发完成：自动预览

完成开发后自动执行：
1. 启动 dev server（如未运行）
2. 在浏览器中打开 `http://localhost:3000/experiments/vN`
3. 等待用户查看效果

### 4. 确认修改

用户下达 **「确认修改」** 指令时，输出结构化变更报告：

```
## 变更报告

| 项目 | 内容 |
|------|------|
| 实验版本号 | vN |
| 基于版本 | vN-1（或 main） |
| 修改点清单 | 1. xxx  2. xxx  3. xxx |
| 文件变更 | 新增: xxx  修改: xxx |
| 构建状态 | ✅ 通过 / ❌ 失败 |
| 与主版本差异 | 摘要说明 |
| 预览地址 | http://localhost:3000/experiments/vN |
```

### 5. 集成主版本

用户确认无误后下达 **「集成主版本」** 指令：

1. 将实验版本 `page.tsx` 内容替换到 `app/page.tsx`
2. 如涉及子组件修改，一并更新到 `components/` 目录
3. 在 `docs/modification-log.md` 记录晋升事件
4. 执行构建验证，确保无报错

### 6. 继续调整

若用户未确认，可下达 **「继续调整」**：
- 在当前实验版本上继续迭代
- 主版本保持不动
- 直到用户满意为止

## 四、子组件隔离策略

### 默认行为
实验版本**共用主版本子组件**（只读引用），不影响主版本稳定性。

### 需要修改子组件时

1. **复制**：将目标子组件复制到实验版本目录
   ```
   components/sections/SocialProof.tsx
   → experiments/v2/components/SocialProof.tsx
   ```

2. **修改**：在副本上进行修改，**不动原文件**

3. **引用**：更新实验版本页面的 import 路径指向本地副本
   ```tsx
   // 修改前（引用共用组件）
   import { SocialProof } from '@/components/sections'
   
   // 修改后（引用本地副本）
   import { SocialProof } from './components/SocialProof'
   ```

### 禁止行为
- ❌ 直接修改 `components/sections/` 等共用组件目录下的文件
- ❌ 在实验版本中引用其他实验版本的子组件

### 变更报告要求
变更报告中需**明确标注**哪些子组件被复制和修改了。

## 五、版本文件头部元数据

每个实验版本文件头部**须包含注释**，格式如下：

```tsx
/**
 * @experiment vN
 * @base vN-1（或 main）
 * @date YYYY-MM-DD
 * @changes 修改摘要
 */
```

**示例**：

```tsx
/**
 * @experiment v2
 * @base v1
 * @date 2026-03-18
 * @changes 优化 Hero 区域动画效果，调整 CTA 按钮样式
 */
```

---

## 六、import 路径调整规则

从 `app/experiments/vN/` 引用共用组件时，需要调整相对路径：

| 原路径（主版本中） | 实验版本中应改为 |
|-------------------|-----------------|
| `@/components/sections/X` | `../../../components/sections/X` |
| `@/components/ui/X` | `../../../components/ui/X` |
| `@/components/illustrations/X` | `../../../components/illustrations/X` |
| `@/lib/X` | `../../../lib/X` |
| `@/hooks/X` | `../../../hooks/X` |

**以下引用保持不变**：
- `next/link`
- `next/image`
- `framer-motion`
- 其他 npm 包

---

## 七、修改日志

每次迭代需更新 `docs/modification-log.md`，格式如下：

| 版本 | 日期 | 修改点 | 备注 |
|------|------|--------|------|
| v1 | 2026-03-18 | 基线版本 | 初始创建 |
| v2 | 2026-03-18 | 优化 Hero 动画 | 待确认 |
| v2 → main | 2026-03-18 | 集成主版本 | 用户确认通过 |

## 八、快速指令参考

| 用户指令 | 执行动作 |
|---------|---------|
| 「新建迭代」 | 检查状态 → 创建新版本 → 开始开发 |
| 「继续调整」 | 在当前实验版本上继续修改 |
| 「确认修改」 | 输出变更报告 |
| 「集成主版本」 | 替换主版本文件 → 更新日志 → 构建验证 |
| 「回滚到 vN」 | 切换到指定历史版本继续迭代 |
